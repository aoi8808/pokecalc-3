# ポケモン ダメージ計算ツール

## 概要
ポケモンの対戦におけるダメージ計算を、ブラウザ上で直感的に行えるWebアプリケーションです。CSVデータを利用して数百匹のポケモンや技のデータを管理し、素早く検索・計算結果を表示できるように設計しました。

## 使用技術
* HTML / CSS
* JavaScript (Vanilla JS)
* データ管理: CSVファイル
* 画像取得: PokeAPI、及び外部サイトからの引用

## こだわったポイント・工夫した点

### 1. 実機プレイとドメイン知識に基づくデータ構築とトライアンドエラー
* **能動的なデータ収集と検証:** 最新作『Pokémon Legends: Z-A』『Pokémon Champions』を自らプレイし、新しく登場するメガシンカポケモンの種族値や、新特性のデータを独自に収集・考察しました。
* **計算精度の向上:** 単にデータを表示するだけでなく、収集した複雑な新特性の補正効果がダメージ計算関数（`calculateDamage`）に正しく乗算されるよう、ゲームの実機画面と見比べながら何度も数値テストとロジックの修正を繰り返しました。

### 2. 外部APIの限界を乗り越える画像表示のデータ設計（仕様変更への柔軟な対応）
* **直面した課題:** 当初は外部の公式API（PokeAPIやPokémon Showdown）から自動で画像を取得する予定でしたが、「日本語名と英語名の名寄せの壁」や、「最新作の画像データが外部サーバーにまだ登録されていない」という致命的な課題に直面しました。
* **アプローチの転換:** そこで外部に依存し続けるのをやめ、自身で用意したGIFアニメーション画像（動くドット絵）をプロジェクト内に一括管理する方式へ切り替えました。
* **データ構造の最適化:** ポケ徹（ポケモン徹底攻略）さんなどの命名規則を参考に、CSVの1列目（ID列）を「画像ファイル名（例: 10278、154m）」と完全に一致させるデータ設計へリファクタリングしました。これにより、プログラム側のコードを肥大化させることなく、CSVの数値を書き換えるだけで新旧すべてのフォルムチェンジ画像が動的に切り替わる、メンテナンス性の高いシステムを実現しました。

### 3. 実践的なデバッグと堅牢なエラーハンドリング
* **多層的なフォールバック処理:** 外部サーバーに画像が存在しない場合や、ローカルでの読み込みに失敗した場合でもアプリがクラッシュしないよう、JavaScriptの `img.onerror` を入れ子構造で配置し、最終的に「画像なし」の代替表示にたどり着く堅牢なエラーハンドリングを実装しました。
* **構造化と開発効率の意識:** `img.src` と `this.src` の役割の違い（イベント発生源のコントロール）を正確に理解してコードに反映しました。また、ファイルの直置きによる煩雑さを防ぐために `image` ディレクトリを切り出して整理し、VS Codeのソース管理（Git）を利用して一括プッシュを行うなど、実際のチーム開発でも通用する効率的な開発プロセスの実践を意識しました。
## ローカルでの動かし方
1. 本リポジトリをクローン、またはダウンロードします。
2. VS Codeの拡張機能「Live Server」等を使用して `index.html` を開きます。
（※ローカルサーバーを立ち上げないと、CSVファイルの読み込み（fetch）でCORSエラーが発生するため）

### 開発における生成AIの活用
本プロジェクトでは、開発効率の向上と技術的な課題解決のために生成AIを活用しています。AIを単なる「コード生成ツール」としてではなく、専属のメンターや壁打ち相手として利用することで、より質の高い実装を目指しました。

* **用途:** エラー発生時の原因究明（デバッグ支援）、JavaScriptの非同期処理やイベントハンドリングに関する壁打ち、リファクタリングの提案。
* **工夫した点:** AIの出力をそのままコピーするのではなく、公式ドキュメントや実際のゲーム仕様と照らし合わせ、最終的な設計判断は自身で行いました。特に画像取得のAPIエラーの際は、AIと議論を重ねて技術的な限界を把握した上で、「ローカルディレクトリでの画像一括管理」という代替案を自ら選択・実装しました。
